# X23 GitHub Pages 靜態預覽版規劃

這份文件規劃的是「可以放到 GitHub Pages 給家人、合作夥伴或妹妹看的靜態預覽版」。它不是正式營運後台，也不能連到真實資料庫、真實帳號或任何可改價、退款、出貨、投廣告的功能。

目標是讓外部觀看者快速看懂：X23 電商營運中台在管什麼、每天要看哪些數字、哪些任務需要人工處理、Agent 分工怎麼協助營運。

## 建議靜態版架構

建議新增一個獨立的 GitHub Pages 目錄，例如：

```text
docs/preview/
  index.html
  assets/
    preview-data.js
    preview.css
    images/
      product-placeholder.jpg
      logo-placeholder.png
```

GitHub Pages 可設定從 `docs/` 發布；若未來不想影響這份說明文件，也可以改成 `docs/preview/index.html` 作為預覽入口。

靜態版應該只使用前端檔案，不依賴：

- `app.py`
- `x23_ops.sqlite3`
- 本機素材資料夾或任何個人電腦上的絕對路徑
- 真實 Notion、Google Sheet、後台或廣告帳號登入狀態
- `POST` API、新增、更新、匯出 CSV 等會改資料的功能

## 預覽版應包含的頁面

靜態預覽版可以沿用 `app.py` 目前的資訊架構，但要改成「展示用假資料」。

### 1. 總覽 Dashboard

用途：讓妹妹一進來就知道這個系統是在管 X23 的整體營運。

建議內容：

- 累計營收、訂單數、估算毛利、廣告花費、ROAS、人工待辦
- 今日重點任務 5 到 8 筆
- 每日營運節奏：早上回填數字、上午分析、下午處理素材/KOL/商品頁、晚上整理明日任務
- Agent 1 到 Agent 4 與人工的分工說明

靜態版只顯示範例數字，例如 `NT$28,200`、`6 筆訂單`，不要使用真實營收或真實毛利。

### 2. 每日銷售

用途：說明每天要怎麼記錄不同通路的銷售狀況。

建議內容：

- 平台：蝦皮、Momo、PChome、官網、寄賣
- 欄位：日期、銷售額、訂單數、退款金額、廣告費、估成本、估毛利、備註
- 一段簡短說明：妹妹只要每天填銷售額、訂單數、廣告費，系統就能幫忙看毛利與 ROAS

靜態版不要提供真正可送出的表單；若要保留表單外觀，按鈕應顯示「預覽不可儲存」。

### 3. 任務分工

用途：讓人看懂哪些事情由 Agent 處理，哪些事情必須人工決定。

建議內容：

- 任務名稱、分類、負責人、優先級、到期日、狀態
- `P0` 顯示為今天或最急，`P1` 顯示為本週，`P2` 顯示為可排程
- 任務分類：帳號/安全、出貨、合規、素材、商品頁、廣告、KOL、數據
- 分工原則：Agent 只做分析、文案、整理、提醒；人工負責帳號登入、出貨、退款、改價、預算核准

任務內容要改成示範任務，不要出現真實合作對象的完整私訊策略或內部敏感備註。

### 4. 通路帳號

用途：說明有哪些銷售與廣告平台需要被管理，但不能洩漏登入資訊。

建議內容：

- 平台名稱：蝦皮、Momo、PChome、官網、Meta Ads、Instagram
- 權限範圍：檢視商品、檢視訂單、檢視廣告數據、草擬文案
- 狀態：待確認、已設定、需人工登入
- 安全提示：靜態版不保存密碼，只保留「密碼放在密碼管理器」的概念

要移除或改寫：

- 真實後台登入網址
- 真實帳號標籤
- `1Password` 或 `Apple Keychain` 中的實際 reference 名稱
- 任何可推測帳號權限或內部管理方式的細節

### 5. 資料源

用途：說明系統會從哪些資料來源整理營運判斷。

建議內容：

- 來源類型：商品頁、銷售報表、廣告報表、KOL 名單、素材資料夾、專案文件
- 狀態：已找到、缺資料、待人工匯出、待確認
- 下一步：匯出報表、補欄位、確認授權、整理素材

要去識別化：

- Notion 頁面網址
- Google Sheet 網址
- 本機檔案絕對路徑
- PDF、Excel 的完整檔名若含內部命名
- 後台或商品頁的私有追蹤參數

公開商品頁網址可視需求保留，但如果預覽版是給非公開對象看，建議改成「公開商品頁 A / B / C」。

### 6. 廣告判讀

用途：讓妹妹看懂「不是投廣告就好，而是要用數字決定加碼或停掉」。

建議內容：

- CTR、CPC、ROAS 的簡單門檻
- 素材狀態：待上線、觀察中、加碼、停掉、重剪
- 範例素材：露營情境版、規格信任版、評價背書版、關鍵字廣告版

要去識別化：

- 真實廣告花費
- 真實營收與購買數
- 未公開素材名稱
- 廣告帳號、像素、受眾設定、投放策略細節

### 7. 貼文排程

用途：展示 IG / FB / Threads 內容如何被排程與追蹤。

建議內容：

- 日期、頻道、主題、格式、素材需求、文案草稿、CTA、負責人、狀態
- 狀態可用：草稿、缺素材、待審核、已排程、已發布

要注意：

- 文案可以保留方向，但不要放未公開的完整貼文稿
- 不要放真實客戶私訊、留言截圖、授權未確認的人物素材
- KOL 或藝人相關內容要用「合作素材 A」、「開箱素材 B」等代稱

### 8. KOL 管線

用途：展示 KOL / 團購合作是如何被分層追蹤。

建議內容：

- 分類：親子露營、露營攝影、團購主、短影音、3C 評測
- 狀態：未聯繫、已聯繫、等回覆、洽談中、已合作
- 合作模式：送品換素材、抽成團購、固定費合作、自然口碑

一定要去識別化：

- 真實 KOL 姓名
- IG / TikTok / Threads handle
- 粉絲數如果會識別個人，改成區間，例如 `5K-10K`
- DM 話術、報價、抽成、合作條件細節
- 任何未取得公開授權的合作備註

### 9. 素材庫

用途：讓人看懂目前有哪些素材類型，以及還缺什麼。

建議內容：

- 素材分類：商品圖、情境圖、短影音、品牌 Logo、社會證明、KOL/人物、規格圖
- 使用位置：官網、蝦皮、Momo、PChome、IG Reels、FB 廣告
- 缺口：白底主圖、完整規格圖、組合包內容圖、15 秒短影音、認證信任圖

要去識別化：

- 本機素材檔名
- 人物照片
- 未授權的評價截圖
- 真實影片檔
- 任何含個資、訂單、後台畫面的圖片

預覽版建議使用 placeholder 圖，或只使用已確認可公開的產品照。

### 10. 出貨交接

用途：說明收到訂單後，人工要怎麼處理出貨。

建議內容：

- 日期、平台、訂單編號、品項、數量、出貨狀態、問題類型、備註
- 問題類型：正常、缺貨、地址問題、客訴、退貨

一定要去識別化：

- 訂單編號
- 收件人姓名
- 電話
- 地址
- 物流單號
- 客訴原文

靜態版的訂單編號可用 `DEMO-ORDER-001` 這種格式。

### 11. 年度計畫

用途：說明 X23 從打底、測試、擴張、旺季到復盤的節奏。

建議內容：

- 期間
- 主題
- 目標
- 關鍵指標
- 負責角色
- 狀態

這頁可以相對完整保留策略框架，但不要放真實營收目標、庫存壓力、合作底價或尚未公開的新品節奏。

### 12. 高風險審核

用途：讓觀看者理解哪些動作不能交給 Agent 自動執行。

建議內容：

- 改價格
- 退款
- 取消訂單
- 下架商品
- 改庫存
- 提高廣告預算

靜態版只展示規則與假紀錄，不提供任何可送出的審核功能。

## 需要去識別化的資料清單

公開到 GitHub Pages 前，至少要處理以下資料：

- 帳號與登入：後台登入網址、帳號名稱、密碼管理器 reference、權限細節
- 財務與銷售：真實營收、真實訂單數、毛利、成本、退款、廣告費、ROAS
- 訂單與客戶：訂單編號、姓名、電話、地址、物流單號、客訴內容
- 合作與 KOL：姓名、handle、報價、抽成、DM 內容、合作進度、未公開備註
- 內部文件：Notion 連結、Google Sheet 連結、本機絕對路徑、PDF/Excel 檔名
- 素材資產：人物照、影片、評價截圖、後台截圖、未授權素材
- 技術資訊：SQLite 檔名、API 路徑若含內部操作、可寫入的端點、伺服器設定

## 妹妹要怎麼看懂

預覽版首頁應該用白話說明：

1. 這不是一個普通網站，而是 X23 的營運駕駛艙。
2. 每天先看總覽：今天有多少訂單、花了多少廣告費、還有幾個人工待辦。
3. 如果數字怪怪的，就看每日銷售與廣告判讀。
4. 如果事情很多，就看任務分工，先做 `P0`，再做 `P1`。
5. 如果有人要登入帳號、退款、改價格、加預算，這些都要人工確認，不能讓系統自己做。
6. KOL、貼文、素材不是分散在聊天紀錄裡，而是集中到各自頁面追蹤。
7. 出貨頁只看今天要寄什麼、有沒有缺貨或地址問題。
8. 年度計畫是大方向，不是每天都要改；每天主要看總覽、任務、銷售、出貨。

建議在靜態版加一個「妹妹版導覽」區塊，用很短的句子呈現：

```text
先看總覽 -> 看 P0 任務 -> 回填銷售 -> 檢查出貨 -> 有高風險就問人工
```

## 靜態資料建議

GitHub Pages 不能使用後端 API，所以資料建議放在 `docs/preview/assets/preview-data.js`：

```js
window.PREVIEW_DATA = {
  summary: {},
  tasks: [],
  sales: [],
  accounts: [],
  dataSources: [],
  ads: [],
  socialPosts: [],
  kols: [],
  assets: [],
  fulfillment: [],
  annualPlan: [],
  approvals: []
};
```

所有資料都要使用 demo 值，例如：

- `DEMO-ORDER-001`
- `KOL A`
- `平台 A`
- `商品頁 A`
- `NT$12,000`
- `2026-05-01`
- `密碼管理器：不公開`

## 功能取捨

靜態預覽版應保留：

- 分頁導覽
- 表格展示
- KPI 卡片
- 任務優先級視覺
- 素材分類
- 簡單篩選或切換狀態，如果只在瀏覽器本機運作

靜態預覽版應移除或改成 disabled：

- 新增銷售
- 新增任務
- 更新任務完成
- 新增出貨
- 更新 KOL 狀態
- 新增貼文
- 高風險審核送出
- 匯出 CSV
- 讀取 `/api/state`
- 讀取 `/media/真實檔名`

## 發布前檢查清單

- [ ] 沒有修改 `app.py`
- [ ] 沒有提交 `x23_ops.sqlite3`
- [ ] 沒有本機絕對路徑
- [ ] 沒有真實帳號、密碼 reference 或後台入口
- [ ] 沒有真實訂單、電話、地址、物流單號
- [ ] 沒有真實 KOL handle、報價或 DM
- [ ] 沒有未授權人物照、影片、評價截圖
- [ ] 所有互動按鈕都不會寫入資料
- [ ] GitHub Pages 首頁能直接開啟，不需要 Python server
- [ ] 妹妹看首頁 30 秒內能知道：這是 X23 營運中台的展示版

## 建議實作順序

1. 先做 `docs/preview/index.html`，複製目前 UI 的分頁架構，但移除所有 `fetch("/api/...")`。
2. 建立 `preview-data.js`，手動放去識別化假資料。
3. 建立 `preview.css`，沿用目前深色中台風格，但讓按鈕與表格更像展示版。
4. 把新增、更新、匯出按鈕改成 disabled 或提示「GitHub Pages 預覽版不可儲存」。
5. 用瀏覽器直接開 `docs/preview/index.html` 測試，不需要啟動 `app.py`。
6. 最後再開 GitHub Pages，確認公開頁面沒有任何敏感資料。
